Security-focused static analysis — often called SAST (Static Application Security Testing) — finds vulnerabilities in source code before they reach production. These tools look for injection flaws, hardcoded secrets, insecure configurations, vulnerable dependencies, and many other classes of security issues.

This page covers the tools tagged with `security` in the catalog, organized by what they scan.

<Note>
  The symbol © marks proprietary tools. ⚠️ means the tool has not been updated in over a year or the repository was archived.
</Note>

<Tip>
  Many tools in this page also appear elsewhere in the catalog — for instance, Semgrep and CodeQL are universal analyzers with strong security rule sets, and Trivy also appears in the containers category. Use the `security` tag on [analysis-tools.dev](https://analysis-tools.dev/) to filter the full list.
</Tip>

### Semgrep for security

Semgrep ships with a large library of security rules covering OWASP Top 10, CWE, and language-specific vulnerability classes. You can run security-focused rulesets without writing any rules yourself:

```bash
semgrep --config "p/security-audit" .
semgrep --config "p/owasp-top-ten" .
```

Semgrep Pro (© commercial) adds inter-file taint analysis and supply chain vulnerability scanning.

### Bearer

Bearer is an open-source static code analysis tool that discovers, filters, and prioritizes security risks and vulnerabilities leading to sensitive data exposures (PII, PHI, PD). It is highly configurable and easily extensible, built for security and engineering teams.

## Secret scanning

Secrets — API keys, passwords, tokens — accidentally committed to source code are one of the most common causes of breaches. These tools scan your git history and file contents to find them.

| Tool | License | Notes |
|------|---------|-------|
| [Gitleaks](https://github.com/zricethezav/gitleaks) | MIT | Detects hardcoded secrets like passwords, API keys, and tokens in git repos |
| [TruffleHog](https://trufflesecurity.com) | AGPL-3.0 | Scans your entire tech stack for exposed credentials |
| [detect-secrets](https://github.com/Yelp/detect-secrets) | Apache-2.0 | Enterprise-friendly secret detection using heuristic regex patterns |
| [Credential Digger](https://github.com/SAP/credential-digger) | Apache-2.0 | GitHub scanning with ML-based false positive filtering |
| [GitGuardian ggshield](https://www.gitguardian.com/ggshield) | MIT | CLI for detecting 350+ types of secrets; runs locally or in CI |

### Gitleaks

Gitleaks is a SAST tool for detecting hardcoded secrets like passwords, API keys, and tokens in git repos. It scans the full commit history and is designed for CI integration:

```bash
# Scan the current repo
gitleaks detect --source . --report-format json --report-path report.json

# Scan a specific git ref range (useful in CI)
gitleaks detect --log-opts="HEAD~10..HEAD"
```

### TruffleHog

TruffleHog is an open-source secret-scanning engine that resolves exposed secrets across your company's entire tech stack. It verifies discovered credentials against live services to identify which secrets are actually valid and exploitable.

## Dependency vulnerability scanning

Your application code may be clean, but the packages you depend on may not be. These tools scan your dependency manifests and lock files against databases of known vulnerabilities.

| Tool | License | Notes |
|------|---------|-------|
| [trivy](https://github.com/aquasecurity/trivy) | Apache-2.0 | Vulnerability scanner for containers, filesystems, repos; covers OS packages and app dependencies |
| [Grype](https://github.com/anchore/grype) | Apache-2.0 | Vulnerability scanner for container images and filesystems; supports SBOM and VEX |
| [OSV-Scanner](https://osv.dev/) | Apache-2.0 | Google's scanner using the OSV.dev database; supports lockfiles, SBOMs, containers |
| [Snyk Code](https://snyk.io) © | Proprietary (free tier) | AI-powered SAST + SCA for Java, JS, Python, PHP, C#, Go, TS |
| [OWASP Dependency Check](https://owasp.org/www-project-dependency-check/) | Apache-2.0 | OWASP project; identifies known CVEs in project dependencies |
| [Safety](https://safetycli.com/) | MIT | Python dependency vulnerability scanner by PyPA |
| [pip-audit](https://github.com/pypa/pip-audit) | Apache-2.0 | Official PyPA tool for scanning Python packages for known vulnerabilities |

### trivy

Trivy is a simple and comprehensive vulnerability scanner for containers and other artifacts, suitable for CI. It detects vulnerabilities of OS packages (Alpine, RHEL, CentOS, etc.) and application dependencies (Bundler, Composer, npm, yarn, etc.) and checks containers and filesystems.

```bash
# Scan a container image
trivy image python:3.12-slim

# Scan the local filesystem
trivy fs --scanners vuln,secret .
```

### OSV-Scanner

OSV-Scanner is written in Go and uses data from the [OSV.dev](https://osv.dev/) open vulnerability database. It scans dependency lockfiles, SBOMs, and container images across multiple languages and package managers for known vulnerabilities. It also supports guided remediation to suggest fix paths.

### bandit

Bandit is a tool for finding common security issues in Python code. It processes each file, builds an AST, and runs appropriate plugins against the AST nodes. It produces a report of all issues found.

```bash
pip install bandit
bandit -r src/
```

### brakeman

Brakeman scans Ruby on Rails applications for security vulnerabilities without requiring a running application. It checks for SQL injection, XSS, CSRF, mass assignment, and dozens of other Rails-specific issues.
