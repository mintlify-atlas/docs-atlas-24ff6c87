Tags are the primary categorization mechanism in the Analysis Tools catalog. Every tool entry specifies one or more tag values in its `tags` field. These values power the language and category filters on [analysis-tools.dev](https://analysis-tools.dev) and group tools in the generated README.

## Tag types

Tags are defined in `data/tags.yml`. Each tag has three properties:

```yaml
- name: Python
  value: python
  type: language
```

| Property | Description |
|----------|-------------|
| `name`   | Human-readable display name shown on the website and in the README |
| `value`  | Slug used in tool YAML files — always lowercase, no spaces |
| `type`   | Either `language` or `other` |

**Language tags** (`type: language`) represent programming languages. Tools tagged with a language tag appear under that language's section in the README and on the website.

**Other tags** (`type: other`) represent frameworks, file formats, tooling categories, platforms, and similar concepts that are not strictly programming languages. Examples include `dockerfile`, `kubernetes`, `security`, and `terraform`.

## Using tags in a tool entry

In `data/tools/<toolname>.yml`, the `tags` field is a list of tag `value` slugs — not display names:

```yaml
tags:
  - python
  - javascript
  - security
  - dockerfile
```

Add every tag that applies. A tool can have any number of tags. The more tags you add, the more filters the tool will appear under on the website, which increases its discoverability.

<Tip>
  When in doubt, add more tags rather than fewer. A Python linter that also supports Django templates should include both `python` and `template`.
</Tip>

## Adding a new tag

If a tool you're adding doesn't fit any existing tag, you can introduce a new one. Add the tag to `data/tags.yml` in the same pull request as the tool entry.

A tag entry in `data/tags.yml` looks like this:

```yaml
- name: WebGPU
  value: webgpu
  type: other
```

Follow these conventions when creating a new tag:

- Choose a `name` that matches how the technology is commonly referred to.
- Choose a `value` that is lowercase, uses hyphens instead of spaces, and is as short as unambiguous.
- Set `type` to `language` for programming languages, or `other` for everything else.
- Place the entry in alphabetical order by `name` within `data/tags.yml`.

<Warning>
  All tag values referenced in tool YAML files must exist in `data/tags.yml`. If you add a tag to a tool entry but forget to add it to `data/tags.yml`, `make render` will fail with an "Invalid tag" error.
</Warning>

After adding a new tag, run `make render` to confirm the tag is recognised and the tool entry validates correctly.
